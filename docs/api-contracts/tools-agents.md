# Tools & Agents Contract

> Shared sources: `shared/tools.ts`, `shared/tools-agents.ts`,
> `shared/canvas-actions.ts`, `shared/connection-matrix.ts`,
> `shared/plan.ts`.

This document records the Agent-ready tool vocabulary used after the manual
assets/workflows/canvas parity pass. It does not enable Agent automation by
itself; it defines the vocabulary future Agents may call once orchestration is
turned on.

Task 60 preflight: `docs/progress/task-60-agent-plan-apply-readiness.md`.

## Phase A acceptance gate

`HDR-PHASEA-001` remains the Phase A acceptance gate. Agent automation over the
migrated workflow vocabulary stays disabled until human review pass or explicit
product deferral.

The future CanvasPlan node vocabulary is limited to text, image, video,
imageConfigV2, videoConfigV2, character, scene, audio, videoCompose,
superResolution, muxAudioVideo. MJ is legacy-known but unavailable for planning,
add paths, run steps, URL refresh, and local automation.

CanvasPlan node vocabulary exact set: text, image, video, imageConfigV2, videoConfigV2, character, scene, audio, videoCompose, superResolution, muxAudioVideo.

The future CanvasPlan edge vocabulary is limited to promptOrder, imageOrder,
imageRole, outputLink, reference, default. Future Agent planning must preserve
clarify branches and dropped warnings for underspecified, unsafe, invalid, or
unavailable requests.

CanvasPlan edge vocabulary exact set: promptOrder, imageOrder, imageRole, outputLink, reference, default.

## Unified Tool Interface

Agents call tools through ToolRuntime descriptors from `shared/tools.ts`.
Manual UI and ToolRuntime graph mutations share `shared/canvas-actions.ts` for
durable create/connect/delete/default-data semantics. Renderer code may keep
undo stacks, transient selection, hover state, drag previews, viewport
animation, and localized feedback copy.

Rules:

- Tools return data, job tickets, or safe errors; never executable code.
- Provider-spending tools return durable job tickets, not generated bytes.
- `tool.list` is the discovery source for enabled tools.
- `tool.invoke` is the execution source for Agents and plugins.
- IPC/service-backed operations remain valid manual equivalents when no
  ToolRuntime wrapper exists yet.
- MJ node/component actions remain out of scope for local Phase A. `mjImage`
  is only a legacy-known graph type.

### Schema Notes

- `canvas.createNode` and `canvas.connectToCreate` accept optional `data`.
  Missing data is filled by `shared/canvas-actions.ts`.
- `canvas.connectNodes` and `canvas.connectToCreate` reject invalid or
  duplicate edges with `ToolError.code = "invalid_edge"`.
- `canvas.runNode` rejects unavailable node definitions before enqueueing.
  MJ is unavailable in local Phase A.
- `canvas.runNode` dispatches generation config nodes by node type:
  `imageConfigV2` enqueues `canvas.generateImage`, while `videoConfigV2`
  enqueues `canvas.generateVideo`.
- `canvas.runNode` resolves reference assets through the workflow asset
  resolver before enqueue. For video jobs, first-frame, last-frame, and
  reference image inputs SHALL be provider-readable cloud URLs when storage is
  configured, with local safe URL fallback only when cloud storage is absent or
  rejected by the host guard.
- `canvas.runNode` may return retryable `job_enqueue_failed` if the durable
  queue cannot persist the job.
- Graph save/load/version/import/export, snippets, workflows, styles, jobs,
  assets, and media edit/drop flows are currently IPC/service-backed
  equivalents documented in `docs/api-contracts/tools-plugins.md`.
- `asset.ensureCloudUrl` returns `{ assetId, url, source, action, s3Key? }`.
  When no storage provider is configured it returns the local safe URL with
  `source = "local"` and `action = "local_fallback"`.

### Permission Model

- `canvas.read`: read graph or validate without mutation.
- `canvas.write`: mutate durable canvas graph state.
- `destructive`: delete graph items; current built-in cleanup tools allow this
  for orchestrator graph edits, while future user-facing Agents may ask.
- `provider.spend`: enqueue gateway/provider-backed work.
- `file.read`: read app-controlled asset bytes for cloud upload.
- `file.write`: reserved for future file/media export tools; current Phase A
  Canvas tools do not require it.
- `network`: call configured storage/provider endpoints.
- Sub-agent effective tools must be a subset of the parent Agent's tools.

### Unsupported Or Manual-Only Actions

- MJ node/component actions, MJ multi-result UI parity, and MJ URL refresh are
  out of scope for local Phase A.
- Seedance/live-person flows are out of scope.
- Transient UI-only actions stay in renderer state: viewport fit, hover menus,
  drag previews, selection rectangle, panel open/close state, and animations.
- Manual-only transient examples include viewport fit, hover menus, drag previews.
- Manual desktop acceptance remains a human review checklist item, not an
  automated Agent action.

### Agent Plan Apply Examples

Create a text prompt node, create an image generation config node, connect
them, then run the generation config node:

```json
[
  {
    "toolId": "canvas.createNode",
    "input": {
      "type": "text",
      "position": { "x": 0, "y": 0 },
      "data": { "label": "Prompt", "content": "rainy neon alley" }
    }
  },
  {
    "toolId": "canvas.createNode",
    "input": {
      "type": "imageConfigV2",
      "position": { "x": 320, "y": 0 }
    }
  },
  {
    "toolId": "canvas.connectNodes",
    "input": {
      "source": "text-node-id",
      "target": "image-config-node-id",
      "edgeType": "promptOrder"
    }
  },
  {
    "toolId": "canvas.runNode",
    "input": { "nodeId": "image-config-node-id" }
  }
]
```

Insert a connected video generation config node from a bound image reference
node. `image` remains a media reference node and is not runnable:

```json
{
  "toolId": "canvas.connectToCreate",
  "input": {
    "source": "image-node-id",
    "type": "videoConfigV2",
    "position": { "x": 640, "y": 0 },
    "edgeType": "imageRole",
    "imageRole": "first_frame"
  }
}
```

## Sub-Agent Tooling

`agent.spawnSubAgent` remains the Agent-level tool for isolated long-running
tasks. Requested tools and skills are intersected with the parent Agent's
permissions; dropped entries are audited. Recursion depth remains capped by
`MAX_SPAWN_DEPTH(2)`.

Safety rules:

- Child Agents cannot gain tools their parent does not hold.
- Child Agents cannot bypass ToolRuntime permission checks.
- Child Agents should prefer `canvas.proposePlan` and `canvas.validateGraph`
  before write tools when intent is ambiguous.
- Provider-spending steps should be explicit and return job tickets only.
