# Jobs Contract

## Owner

- Primary: tooling-agent
- Supporting: pm-agent
- Shared source: `shared/jobs.ts`, `shared/ipc.ts`

## Scope

This contract defines the durable local JobRuntime used for generation, gateway calls, agent runs, indexing, plugin reloads, and other slow operations. Every slow operation SHALL enqueue a persisted job before returning to the renderer.

Non-goals:

- No Redis, BullMQ, or remote queue.
- No WebSocket dependency; terminal updates use Electron IPC events.
- No synchronous model/provider execution inside request handlers.

## Request/Response Contracts

### `job.enqueue`

Request:

```ts
interface JobCreateInput {
  type: JobType
  targetId?: string
  payload: Record<string, unknown>
  requestedBy: JobActor
  idempotencyKey?: string
}
```

Response:

```ts
interface JobTicket {
  jobId: string
  status: 'pending'
  createdAt: number
}
```

### `job.get`

Request:

```ts
interface JobGetRequest {
  jobId: string
}
```

Response:

```ts
interface JobRecord {
  id: string
  type: JobType
  status: JobStatus
  targetId?: string
  progress: number
  result?: JobResult
  error?: JobError
  createdAt: number
  updatedAt: number
}
```

### Terminal Events

Events:

```ts
type JobTerminalEvent =
  | { channel: 'job.completed'; jobId: string; result: JobResult; emittedAt: number }
  | { channel: 'job.failed'; jobId: string; error: JobError; emittedAt: number }
```

Rules:

- Valid states: `pending -> processing -> completed | failed | canceled`.
- For each job ID, exactly one terminal event SHALL be emitted.
- Terminal result SHALL be persisted before terminal IPC emission.
- Startup recovery SHALL reconcile stale `processing` jobs before accepting worker work.

## Errors

| Error class | Meaning |
| :--- | :--- |
| `job_not_found` | No job exists for the requested ID. |
| `job_payload_invalid` | The payload failed schema validation. |
| `job_transition_invalid` | Runtime attempted an illegal state transition. |
| `job_worker_interrupted` | Startup recovery found an abandoned worker lease. |
| `job_retry_exhausted` | Retry policy was exhausted. |

## Permissions

- Any UI or agent call may enqueue only job types allowed by its capability policy.
- Provider-spending jobs require gateway permission policy.
- Job cancellation and retry are write actions and require explicit ownership or elevated permission.

## Tests

- Unit: state-machine transition table.
- Unit: duplicate terminal emission is rejected.
- Integration: enqueue returns only ticket data.
- Integration: stale `processing` recovery policy is deterministic.
- Static/deep scan: synchronous responses do not contain generated bytes, absolute paths, safe URLs, or provider temporary URLs.
