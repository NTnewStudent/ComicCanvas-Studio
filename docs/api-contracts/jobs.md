# Jobs Contract

## Owner

- Primary: tooling-agent
- Supporting: pm-agent
- Shared source: `shared/jobs.ts`, `shared/ipc.ts`

## Scope

This contract defines the durable local JobRuntime used for generation, gateway calls, agent runs, indexing, plugin reloads, and other slow operations. Every slow operation SHALL enqueue a persisted job before returning to the renderer.

Non-goals:

- No Redis, BullMQ, or remote queue.
- No WebSocket dependency; terminal updates use Electron IPC events.
- No synchronous model/provider execution inside request handlers.

## Request/Response Contracts

### `job.enqueue`

Request:

```ts
interface JobCreateInput {
  type: JobType
  targetId?: string
  payload: Record<string, unknown>
  requestedBy: JobActor
  idempotencyKey?: string
}
```

Response:

```ts
interface JobTicket {
  jobId: string
  status: 'pending'
  createdAt: number
}
```

Canvas job types:

- `canvas.generateImage`, `canvas.generateVideo`, `canvas.generateAudio`,
  `canvas.composeVideo`, `canvas.upscaleVideo`, and `canvas.muxAudioVideo`
  SHALL use asset/report terminal results and SHALL NOT synchronously return
  generated bytes or URLs from enqueue IPC.
- `canvas.polishText` SHALL be the local text polish job type for CanvasPlan
  `textPolish` and manual TextNode AI polish. Its terminal success result
  SHALL be `{ kind: 'text', text }`.

### `job.get`

Request:

```ts
interface JobGetRequest {
  jobId: string
}
```

Response:

```ts
interface JobRecord {
  id: string
  type: JobType
  status: JobStatus
  targetId?: string
  progress: number
  result?: JobResult
  error?: JobError
  createdAt: number
  updatedAt: number
}
```

### Terminal Events

Events:

```ts
type JobTerminalEvent =
  | { channel: 'job.completed'; jobId: string; result: JobResult; emittedAt: number }
  | { channel: 'job.failed'; jobId: string; error: JobError; emittedAt: number }

interface JobProgressEvent {
  channel: 'job.progress'
  jobId: string
  progress: number
  message?: string
  emittedAt: number
}
```

Rules:

- Valid states: `pending -> processing -> completed | failed | canceled`.
- For each job ID, exactly one terminal event SHALL be emitted.
- Terminal result SHALL be persisted before terminal IPC emission.
- Progress events MAY be emitted while a job is processing and SHALL NOT count as terminal events.
- Startup recovery SHALL reconcile stale `processing` jobs before accepting worker work.
- Main-process fanout SHALL send terminal events over the whitelisted Electron IPC channel matching `event.channel`.
- The preload bridge SHALL expose typed subscription helpers for `job.completed` and `job.failed`, returning unsubscribe callbacks without exposing raw `ipcRenderer`.
- Renderer consumers SHALL use IPC event subscriptions plus query invalidation for terminal job state, not `setInterval`, asset polling loops, or TanStack Query `refetchInterval`.

### Renderer Reopen Reconciliation

Rules:

- When a workflow graph is loaded, the renderer SHALL query recent jobs once
  through `job.list` and reconcile missed terminal or active generation state.
- Reconciliation SHALL cover non-MJ generation jobs for image, video,
  character, scene, audio, videoCompose, superResolution, and muxAudioVideo
  nodes, plus `canvas.polishText` jobs for text nodes.
- Completed asset/report job results SHALL patch node `status`, `assetId`,
  `url`, `urls`, and `selectedIndex` where present.
- Generation, compose, mux, upscale, and audio job payloads SHALL resolve
  asset references through the workflow asset resolver before enqueueing:
  local `cc-asset://` fallback is always valid, cloud URLs are refreshed or
  re-signed only when storage is configured and the refreshed host matches the
  configured endpoint or `publicUrlPrefix`.
- Completed text polish results SHALL patch text node `content`, `html`, and
  `polishStatus: 'done'`.
- Failed jobs SHALL patch node `status: 'error'` and preserve the safe error
  message for recovery UI.
- Failed text polish jobs SHALL patch `polishStatus: 'error'` and preserve the
  safe error message for recovery UI.
- Pending or processing jobs SHALL patch node `status: 'pending'` and clear
  stale `assetId`.
- Pending or processing text polish jobs SHALL patch `polishStatus` to
  `pending` or `running` without touching asset fields.
- The renderer MAY surface a one-shot recovery summary after graph load, but it
  SHALL NOT poll for job state.

### Renderer Run History Panel

Rules:

- The canvas run panel SHALL render recent jobs from `job.list` as a local run
  history list without polling.
- Each run history row SHALL expose job type, status, target node, progress, and
  safe error text when available.
- Selecting a row SHALL show a detail view with per-node target, progress,
  terminal output summary, failed/canceled state, and error class/message.
- Manual rerun SHALL call the existing `canvas.runNode`/preload run surface for
  the selected target node; it SHALL NOT introduce a separate rerun IPC channel.
- Terminal job events MAY refresh the list through subscriptions and SHALL NOT
  require a renderer `setInterval`.

## Errors

| Error class | Meaning |
| :--- | :--- |
| `job_not_found` | No job exists for the requested ID. |
| `job_payload_invalid` | The payload failed schema validation. |
| `job_transition_invalid` | Runtime attempted an illegal state transition. |
| `job_worker_interrupted` | Startup recovery found an abandoned worker lease. |
| `job_retry_exhausted` | Retry policy was exhausted. |

## Permissions

- Any UI or agent call may enqueue only job types allowed by its capability policy.
- Provider-spending jobs require gateway permission policy.
- Job cancellation and retry are write actions and require explicit ownership or elevated permission.

## Tests

- Unit: state-machine transition table.
- Unit: duplicate terminal emission is rejected.
- Integration: enqueue returns only ticket data.
- Integration: stale `processing` recovery policy is deterministic.
- Integration: terminal events fan out to live renderer windows exactly once.
- Renderer: job terminal IPC events invalidate job and asset queries without polling.
- Static/deep scan: synchronous responses do not contain generated bytes, absolute paths, safe URLs, or provider temporary URLs.
